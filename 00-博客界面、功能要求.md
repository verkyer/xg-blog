# 博客界面、功能要求

本文档用于约束小鸽志静态博客的公开前台界面、布局结构和文章阅读体验。本文档只描述博客展示层需求，不包含本地编辑器、评论、阅读统计、点赞等动态功能。

## 1. 页面目标

博客前台需要保持内容站、技术博客和优惠信息站的阅读效率，页面结构应清晰、稳定、利于 SEO，并适合长期维护。

核心目标：

- 首页采用双栏博客布局。
- 顶栏显示站点 logo 和分类导航。
- 文章列表使用左图右文的信息结构。
- 右侧栏提供作者介绍、分类目录和标签入口。
- 文章内页左侧显示正文，右侧显示悬浮文章目录。
- Markdown 正文需要具备良好的默认排版、图片灯箱、代码高亮和代码复制能力。
- 页面需要适配桌面端、平板端和移动端。

## 2. 整体布局要求

### 2.1 页面宽度

桌面端建议使用居中容器：

```text
max-width: 1200px ~ 1280px
```

主体内容使用双栏布局：

```text
左侧主内容区：约 70%
右侧侧边栏：约 30%
```

建议宽度：

```text
主内容区：760px ~ 860px
侧边栏：280px ~ 340px
栏间距：24px ~ 32px
```

### 2.2 响应式要求

桌面端：

- 顶栏固定在页面顶部或保持普通文档流均可。
- 主体使用左右双栏。
- 右侧栏展示作者卡片、分类目录、标签。
- 文章内页右侧文章目录可以 sticky 悬浮。

平板端：

- 主体可继续双栏，也可缩窄右侧栏。
- 如果空间不足，右侧栏应移动到主内容下方。

移动端：

- 顶栏需要适配小屏，分类导航可横向滚动或折叠菜单。
- 主体改为单栏。
- 首页文章卡片从左图右文改为上图下文，或使用小缩略图加右侧文字。
- 文章内页目录不再右侧悬浮，可收起为正文上方目录或隐藏。

## 3. 顶栏要求

顶栏至少包含：

- logo。
- 站点名称或 logo 文案。
- 分类导航。

分类导航数据来源：

```text
site/src/content/data/categories.yml
```

导航展示要求：

- 优先显示常用分类。
- 分类项链接到 `/categories/{categorySlug}`。
- 当前分类页应有激活态。
- 分类数量较多时，不要求全部塞进顶栏，可保留主要分类，其余通过分类页或更多菜单访问。

## 4. 首页双栏布局

### 4.1 左侧文章列表

首页左侧为文章列表。每篇文章使用左图右文结构：

```text
┌──────────────┬────────────────────────────┐
│ 文章封面图   │ 第一排：文章标题             │
│              │ 第二排：SEO 信息 / 描述      │
│              │ 第三排：日期 + 分类 + 标签   │
└──────────────┴────────────────────────────┘
```

文章列表项字段：

- 封面图。
- 标题。
- SEO 描述。
- 发布日期。
- 主分类。
- 标签。
- 文章链接。

字段来源建议：

```yaml
title: 文章标题
slug: article-slug
date: 2026-05-16
category: vps
tags:
  - cloudflare
  - vps
cover: ./img/cover.webp
description: 用于 SEO 和列表展示的摘要
```

展示要求：

- 标题位于第一排，最多显示 2 行。
- SEO 信息也作为文章描述，位于第二排，最多显示 2 ~ 3 行。
- 日期、分类、标签位于第三排。
- 分类和标签需要可点击。
- 封面图需要固定比例，建议使用 `16:9` 或 `4:3`。
- 没有封面图时使用默认占位图或站点 logo 占位。
- 文章列表按发布时间倒序展示。
- 置顶文章如已有权重字段，应优先展示。

### 4.2 右侧作者介绍卡片

首页右侧顶部为作者介绍卡片。

卡片内容：

- logo 或头像。
- 昵称。
- 简短描述。

建议数据来源：

```text
site/src/content/data/site.yml
```

示例字段：

```yaml
author:
  name: 小鸽
  avatar: /logo.svg
  description: 记录 VPS、建站、运维和互联网工具。
```

展示要求：

- logo / 头像清晰显示。
- 昵称需要醒目但不过度放大。
- 描述控制在 1 ~ 3 行。
- 卡片可包含关于页面链接，但不是必须。

### 4.3 右侧分类目录

作者卡片下方为分类目录。

展示要求：

- 默认显示 8 个分类。
- 每个分类显示分类名称。
- 如果能拿到文章数量，可显示文章数量。
- 超过 8 个分类时显示“更多”入口。
- “更多”入口链接到 `/categories`。

分类链接格式：

```text
/categories/{categorySlug}
```

### 4.4 右侧标签区域

分类目录下方为标签区域。

展示要求：

- 默认显示 20 个标签。
- 标签需要可点击。
- 如果能拿到文章数量，可按文章数量或配置顺序排序。
- 超过 20 个标签时显示“更多”入口。
- “更多”入口链接到 `/tags`。

标签链接格式：

```text
/tags/{tagSlug}
```

## 5. 文章内页布局

### 5.1 页面结构

文章内页采用左正文、右目录布局：

```text
┌──────────────────────────────┬──────────────────┐
│ 文章标题                      │ 文章目录          │
│ 日期 + 分类 + 标签            │ sticky 悬浮       │
│ 文章正文                      │                  │
└──────────────────────────────┴──────────────────┘
```

正文区至少包含：

- 文章标题。
- 发布日期。
- 分类。
- 标签。
- SEO 描述，可作为导语显示，也可只用于 meta。
- Markdown 渲染后的正文。

右侧目录：

- 从正文标题自动生成。
- 建议包含 `h2`、`h3`。
- 当前阅读位置可高亮。
- 桌面端使用 `position: sticky`。
- 移动端隐藏或改为正文顶部折叠目录。

### 5.2 URL 要求

文章内页继续使用既有路由：

```text
/archives/{slug}
```

不得改为：

```text
/posts/{slug}
/blog/{slug}
/{year}/{month}/{slug}
```

## 6. Markdown 正文渲染要求

### 6.1 基础渲染能力

文章正文需要支持标准 Markdown：

- 标题。
- 段落。
- 加粗、斜体、删除线。
- 有序列表、无序列表。
- 引用块。
- 表格。
- 分割线。
- 行内代码。
- 代码块。
- 图片。
- 链接。

如需要更复杂组件，可允许使用 MDX，但普通文章应优先使用 Markdown。

### 6.2 推荐样式方案

如果项目使用 Tailwind CSS，推荐使用：

```text
@tailwindcss/typography
```

原因：

- 已有成熟的 Markdown 排版默认样式。
- 易于统一标题、段落、列表、表格、引用和代码块样式。
- 可通过 `prose` 类快速应用文章排版。
- 后续可覆盖局部 CSS 以适配站点视觉。

如果不使用 Tailwind CSS，推荐使用：

```text
github-markdown-css
```

或自定义一份文章正文样式：

```text
.markdown-body
```

自定义样式至少需要覆盖：

- 标题层级间距。
- 段落行高。
- 列表缩进。
- 表格横向滚动。
- 引用块样式。
- 图片最大宽度。
- 行内代码。
- 代码块。

### 6.3 代码块高亮

Astro 项目中优先使用 Shiki 高亮。

推荐配置：

```text
Astro Markdown 内置 Shiki
```

或：

```text
rehype-pretty-code
```

要求：

- 代码块显示语言类型。
- 代码块支持深色或浅色主题，需和站点主题一致。
- 代码过长时横向滚动，不得撑破页面。
- 代码块右上角提供复制按钮。
- 点击复制后需要有成功反馈。

### 6.4 代码复制

代码复制可以通过前端脚本增强实现。

实现要求：

- 每个代码块自动生成复制按钮。
- 复制内容应为纯代码文本，不包含行号、按钮文案、语言名称。
- 复制成功后显示“已复制”状态。
- 复制失败时不应影响文章阅读。

### 6.5 图片灯箱

正文图片需要支持点击打开灯箱。

推荐方案：

```text
PhotoSwipe
```

或：

```text
medium-zoom
```

选择建议：

- 如果只需要单图放大，使用 `medium-zoom` 即可。
- 如果需要组图浏览、上一张/下一张、移动端手势，使用 `PhotoSwipe`。

灯箱要求：

- 正文图片点击后可放大查看。
- 支持关闭。
- 移动端可正常查看。
- 不影响图片的原始 alt 文本。
- 图片加载失败时不应影响正文布局。

### 6.6 外链和图片 SEO

正文链接要求：

- 外链建议增加 `target="_blank"`。
- 外链建议增加 `rel="noopener noreferrer"`。

图片要求：

- 尽量填写 `alt`。
- 文章封面图和正文关键图片应使用压缩后的 WebP 或 AVIF。
- 图片宽高尽量明确，减少布局偏移。

## 7. 分类页和标签页

分类页：

```text
/categories
/categories/{categorySlug}
```

标签页：

```text
/tags
/tags/{tagSlug}
```

要求：

- 集合页展示全部分类或全部标签。
- 详情页展示对应分类或标签下的文章列表。
- 文章列表样式应尽量复用首页左侧文章列表。
- 分类和标签不存在时返回 404 页面。

## 8. 视觉和交互要求

整体风格：

- 内容优先。
- 信息密度适中。
- 避免过度装饰。
- 保持清晰的阅读层级。

基础要求：

- 页面背景、正文、卡片、边框、链接颜色需要有稳定规范。
- 卡片圆角不宜过大，建议 `6px ~ 8px`。
- 标题、正文、辅助信息的字号层级要明确。
- 日期、分类、标签属于辅助信息，不应抢占标题视觉权重。
- 交互态需要包含 hover、active、focus。

## 9. SEO 要求

首页、分类页、标签页、文章页均需要生成基础 SEO 信息。

文章页至少包含：

- `<title>`。
- `meta description`。
- canonical URL。
- Open Graph 标题。
- Open Graph 描述。
- Open Graph 图片。
- 发布时间。
- 分类和标签结构化信息可按需添加。

首页文章列表中的 SEO 描述应与文章 frontmatter 的 `description` 字段一致。

## 10. 可访问性要求

基础要求：

- logo 图片需要 alt。
- 文章封面图需要 alt。
- 分类和标签链接需要可聚焦。
- 代码复制按钮需要可键盘操作。
- 灯箱打开后需要支持键盘关闭。
- 页面文字和背景需要有足够对比度。
- 移动端点击区域不得过小。

## 11. 暂不包含范围

本文档不包含：

- 评论系统。
- 阅读统计。
- 点赞功能。
- 登录系统。
- 后台管理。
- 在线编辑器。
- 数据库。
- 搜索接口服务。

如后续需要站内搜索，可另行设计静态索引方案，例如 Pagefind。
